# 技術ドキュメント評価基準

本書は `docs/next/` 配下のドキュメントを評価するためのプロトコル。
人間のレビュアーにもAIエージェントにも使える設計にしている。

## 理論的背景

本基準は以下の業界標準フレームワークを統合して策定:

| フレームワーク | 提供する視点 | 本基準での反映箇所 |
|---|---|---|
| **Diataxis** (diataxis.fr) | ドキュメントの4象限分類: Tutorial / How-to / Reference / Explanation | カテゴリ A（構造）, E（リファレンス） |
| **Google Developer Documentation Style Guide** | 文章品質、明確さ、アクセシビリティ、トーン | カテゴリ C（文章品質・アクセシビリティ） |
| **Write the Docs (ARID原則)** | 適度な繰り返し許容、単一情報源、Docs-as-Code | カテゴリ F（保守性・フィードバック機構） |
| **Plain Language Guidelines** | 平易な言語、短文、能動態 | カテゴリ C（明確さ） |
| **ISO/IEC/IEEE 26514:2022** | ユーザー向け情報の設計・開発標準。アクセシビリティ、正確性、一貫性 | カテゴリ C（アクセシビリティ）, B（正確性） |
| **DORA (DevOps Research and Assessment)** | ドキュメント品質の8属性。Time-to-First-Success がパフォーマンスに直結 | カテゴリ D（オンボーディング効果） |

---

## 採点方法

各基準を **0-3** で採点する。

| スコア | 意味 | レビュアーへの指示 |
|--------|------|-------------------|
| **0** | 未対応 / 根本的に欠落 | 該当する内容が存在しない、または機能していない |
| **1** | 存在するが品質が低い | 内容はあるが、誤り・不足・不統一が目立つ |
| **2** | 基本的な期待を満たす | 実用上の問題はないが、改善の余地がある |
| **3** | ベストプラクティス準拠 | 模範的。他のドキュメントの手本になる |

**重要**: スコア3は「完璧」ではなく「十分に良い」。過度な完璧主義は不要。

## カテゴリ A: 構造と構成

> 根拠: Diataxis フレームワーク -- ドキュメントは「学ぶ」「やる」「調べる」「理解する」の4目的に応じて構造化すべき

### A1. コンテンツタイプの分離 (Diataxis 4象限)

**チェック方法**:
1. 各機能ドキュメント（features/*.md）を開く
2. 以下の3タイプが混在せず、セクションで分離されているか確認:
   - **リファレンス**: 型定義、API仕様、テーブル定義（「何があるか」の事実列挙）
   - **ハウツー**: 「よくある操作の流れ」セクション（「こうやる」の手順）
   - **説明**: 概要、ユースケース、ロジック説明（「なぜそうなっているか」）
3. 1つのセクション内で「APIの仕様を説明しながら操作手順も書いている」ような混在がないか

**このプロジェクトでの具体例**:
- 良い例: `needs.md` の「Server Actions」（リファレンス）と「よくある操作の流れ」（ハウツー）が分離
- 悪い例: リファレンスセクション内で「まずこれをして、次にこれをする」のような手順が混入

| スコア | 判定基準 |
|--------|----------|
| 0 | 全てが1つの文書に混在。セクション分けなし |
| 1 | セクション分けはあるが、リファレンス内にハウツーが混在する箇所がある |
| 2 | 明確なセクション分け。時折混在するが全体として良好 |
| 3 | リファレンス/ハウツー/説明が明確に分離。読者は目的に応じたセクションに直行可能 |

### A2. ナビゲーションと発見性

**チェック方法**:
1. `index.md` の「ドキュメント一覧」テーブルが全ファイルをカバーしているか
2. 各ドキュメントの末尾に「関連機能」セクションがあるか
3. あるドキュメントから別の関連ドキュメントに2クリック以内で到達できるか
4. 孤立ページ（どこからもリンクされていないページ）がないか

**チェックリスト**:
- [ ] `index.md` から全ドキュメントへのリンクがある
- [ ] 各 feature doc から関連 feature doc へのリンクがある
- [ ] entity doc から対応する feature doc へのリンクがある
- [ ] `_template.md` への言及またはリンクがある

| スコア | 判定基準 |
|--------|----------|
| 0 | 目次なし、リンクなし |
| 1 | `index.md` に一覧はあるが、ドキュメント間の相互リンクが不足 |
| 2 | 階層的な目次 + 各ドキュメント末尾の「関連機能」リンク |
| 3 | 複数の発見経路: 目次 + 相互リンク + インラインリンク。孤立ページなし |

### A3. 情報アーキテクチャ

**チェック方法**:
1. ドキュメントのディレクトリ構造と `src/` のディレクトリ構造を比較
2. FSD レイヤー（app, views, widgets, features, entities, shared, lib, core）ごとにドキュメントがあるか
3. 「このコードのドキュメントはどこ？」と聞かれたとき、ディレクトリ構造から推測可能か

**期待される対応関係**:
```
docs/next/app/          <-->  app/
docs/next/src/core/     <-->  src/core/
docs/next/src/lib/      <-->  src/lib/
docs/next/src/entities/ <-->  src/entities/
docs/next/src/features/ <-->  src/features/
docs/next/src/shared/   <-->  src/shared/
docs/next/src/views/    <-->  src/views/
docs/next/src/widgets/  <-->  src/widgets/
```

| スコア | 判定基準 |
|--------|----------|
| 0 | ドキュメント構造とコード構造に関連性がない |
| 1 | 一部対応しているが、欠落するレイヤーがある |
| 2 | 全レイヤーにドキュメントがあり、論理的にグルーピング |
| 3 | コード構造と1:1対応。ドキュメントパスからコードパスを推測可能 |

### A4. 段階的詳細化（Progressive Disclosure）

**チェック方法**:
1. `index.md` → レイヤー別 index → 個別ドキュメント の3段階で情報が深まるか
2. 各ドキュメントの冒頭に「概要」があり、詳細は下方にあるか
3. 読者が必要な深さで読み止められるか（概要だけ読んで十分なケースがあるか）

**3段階の期待**:
- **L1 (index.md)**: 全体像。「何ができるシステムか」「どういう技術か」
- **L2 (features/index.md 等)**: レイヤーの役割。「どんな feature があるか」
- **L3 (features/needs.md 等)**: 個別の詳細。「この feature の API は何か」

| スコア | 判定基準 |
|--------|----------|
| 0 | 全詳細が一度に提示される |
| 1 | 一部にネストがあるが一貫性がない |
| 2 | 概要 → 詳細のパターンが使われている |
| 3 | L1/L2/L3 の3段階が明確。各段階に「もっと詳しく」への導線あり |

### A5. 相互参照

**チェック方法**:
1. 機能間の関連（例: needs → ideas, chat-consultation → needs/ideas/solutions）がリンクで表現されているか
2. リンクが双方向か（A→B があるなら B→A もあるか）
3. デッドリンク（存在しないファイルへのリンク）がないか

**主要な関連の期待**:
- needs <-> ideas（ニーズからネタ作成）
- needs <-> search（検索機能）
- ideas <-> search
- chat-consultation <-> needs, ideas, solutions, sources（検索対象）
- auth <-> admin（認可）
- analytics <-> admin（分析表示）

| スコア | 判定基準 |
|--------|----------|
| 0 | 相互リンクなし |
| 1 | 一部にリンクがあるが、多くの関連がリンクされていない |
| 2 | 主要な関連にリンクあり。「関連機能」セクションが大半のドキュメントにある |
| 3 | 双方向リンク完備。文中のインラインリンク + 末尾の「関連機能」セクション |

## カテゴリ B: コンテンツ品質

> 根拠: Write the Docs ARID 原則 -- 正確で最新の情報を、適切な粒度で提供する

### B1. 正確性

**チェック方法**:
1. ドキュメントに記載されたバージョン番号を `package.json` と照合
2. Server Actions の入出力型を実際のコードと照合（最低3つ抽出）
3. DB テーブルのカラム定義を `prisma/schema.prisma` と照合
4. ファイルパスが実際に存在するか確認（最低5つ抽出）
5. コード例の import パスが正しいか確認

**特に注意すべき箇所**:
- `index.md` の技術スタック（Next.js のバージョン等）
- `routes.md` の HTTP メソッド（GET/POST/PATCH/DELETE）
- 各 feature doc の Server Actions 入出力型

| スコア | 判定基準 |
|--------|----------|
| 0 | バージョンや型定義に事実誤認がある |
| 1 | 概ね正確だが、一部のファイルパスやメソッド名が古い |
| 2 | 正確。軽微なズレのみ（例: オプショナルなフィールドの過不足） |
| 3 | 全ての型、パス、バージョンが現在のコードベースと一致 |

### B2. 網羅性

**チェック方法**:
1. `src/features/` のサブディレクトリ一覧を取得
2. 各 feature にドキュメントまたは言及があるか確認
3. 特に分割 feature（need-favorites, need-embedding, idea-details 等）が文書化されているか
4. 全ての API Route (`app/api/**`) がドキュメントに記載されているか
5. Jotai atoms が状態管理セクションで網羅されているか

**チェックリスト（分割 feature）**:
- [ ] need-favorites
- [ ] need-embedding
- [ ] bulk-idea-creation
- [ ] idea-details
- [ ] idea-need-linking
- [ ] dify-idea-update
- [ ] sub-ideas

| スコア | 判定基準 |
|--------|----------|
| 0 | 主要機能（needs/ideas/auth 等）が文書化されていない |
| 1 | コア機能は文書化されているが、分割 feature の説明が不足 |
| 2 | 全機能が基本レベルで文書化。分割 feature も言及あり |
| 3 | 全機能の API/UI/状態管理が文書化。エッジケースやエラーも記載 |

### B3. 鮮度（Currency）

**チェック方法**:
1. `git log --oneline -20` の直近変更が docs に反映されているか
2. FSD 移行後の構造（entities/ui/core-*-table 等）が正しく反映されているか
3. 「@deprecated」と記載された機能が残っていないか
4. 最新の feature（solutions の UI 改善等）が反映されているか

| スコア | 判定基準 |
|--------|----------|
| 0 | FSD 移行前の構造を反映。大幅に古い |
| 1 | FSD 移行は反映されているが、直近の変更が未反映 |
| 2 | 概ね最新。軽微な遅れのみ |
| 3 | 直近のコミットまで反映。Docs-as-Code プラクティスが機能している |

### B4. コード例

**チェック方法**:
1. 各 feature doc にコード例があるか
2. import パスが `@/features/...` 形式で正しいか
3. 入力/出力の型が TypeScript 型で明示されているか
4. コード例をコピーしてそのまま使えるか（必要な import が含まれているか）

**良いコード例の条件**:
- import 文がある
- 入力の型が明示されている
- 出力の型が明示されている（成功/失敗の両方）
- `ctx.tenantId` 等のコンテキスト変数の出所が分かる

| スコア | 判定基準 |
|--------|----------|
| 0 | コード例なし |
| 1 | コード例はあるが import が欠落、または型が不明 |
| 2 | 一般的なケースの動作するコード例。入力/出力型あり |
| 3 | コピー可能な例 + 成功/失敗の両パターン + コンテキストの説明 |

### B5. エラー・トラブルシュート

**チェック方法**:
1. `index.md` にエラー階層（ActionError のサブクラス）が記載されているか
2. 各 feature doc でエラーケース（認証失敗、テナント不一致等）が記載されているか
3. 「こうなったらこうする」のトラブルシュート情報があるか

**このプロジェクトの主要エラーパターン**:
- UnauthorizedError → /sign-in にリダイレクト
- ForbiddenError → テナント不一致
- NotFoundError → リソースが見つからない
- ValidationError → Zod バリデーション失敗

| スコア | 判定基準 |
|--------|----------|
| 0 | エラーに関する記述なし |
| 1 | エラークラスの列挙のみ |
| 2 | エラーの原因と対処法が index.md または feature doc に記載 |
| 3 | エラー原因 + 対処法 + 予防策。各 feature doc で具体的なエラーシナリオを記載 |

## カテゴリ C: 文章品質

> 根拠: Google Developer Documentation Style Guide -- 簡潔で明確、一貫した文体

### C1. 明確さと簡潔さ

**チェック方法**:
1. 各ドキュメントの冒頭1-2文で機能の目的が分かるか
2. 専門用語（FSD, Jotai, pgvector, RRF 等）が初出時に説明されているか
3. 不要な修飾語（「基本的に」「一般的に」「簡単に」）が多用されていないか
4. 1文が長すぎないか（目安: 日本語で50文字以内）

| スコア | 判定基準 |
|--------|----------|
| 0 | 専門用語が未説明。冗長で理解困難 |
| 1 | 一部不明瞭なセクションがある |
| 2 | 概ね明確。冒頭で目的が分かる。時折冗長 |
| 3 | 簡潔で正確。専門用語は初出時に定義。各セクションの目的が一目瞭然 |

### C2. 一貫した文体

**チェック方法**:
1. 全ドキュメントで敬体（です・ます）/常体（だ・である）が統一されているか
2. セクション見出しのパターンが統一されているか（例: 全 feature doc に「概要」「ルーティング」「データモデル」「Server Actions」...がある）
3. テーブルの列名パターンが統一されているか
4. コードブロックの言語指定が統一されているか（typescript, mermaid 等）

| スコア | 判定基準 |
|--------|----------|
| 0 | ドキュメント間で文体がバラバラ |
| 1 | 大半は統一されているが、一部のドキュメントで文体が異なる |
| 2 | 文体は統一。セクション構成に軽微な不統一 |
| 3 | 文体 + セクション構成 + テーブル形式が全ドキュメントで統一。`_template.md` に準拠 |

### C3. 実行可能な指示

**チェック方法**:
1. 「よくある操作の流れ」セクションの手順が番号付きか
2. 各手順に「何を」「どこで」「結果は何か」が書かれているか
3. 前提条件（認証済み、テナント選択済み等）が明示されているか

| スコア | 判定基準 |
|--------|----------|
| 0 | 操作手順がない |
| 1 | 手順はあるが前提条件が不明 |
| 2 | 番号付きステップで主要フローを記載 |
| 3 | 番号付きステップ + 前提条件 + 期待される結果 |

### C4. フォーマットの一貫性

**チェック方法**:
1. 見出し階層: H1（#）はファイルに1つ、H2（##）で大セクション、H3（###）で小セクション
2. コードブロック: 言語指定あり（```typescript, ```mermaid）
3. テーブル: ヘッダー行 + 区切り行 + データ行
4. リスト: `-` で統一（`*` や `+` との混在なし）

| スコア | 判定基準 |
|--------|----------|
| 0 | フォーマットに一貫性なし |
| 1 | 見出しレベルの飛び（H2 → H4）やリスト記号の混在 |
| 2 | セクション内では統一。ドキュメント間で軽微な差異 |
| 3 | 全ドキュメントで統一。Markdown lint 準拠 |

### C5. 用語の一貫性

**チェック方法**:
以下の用語が全ドキュメントで統一されているか:

| 正式名称 | NG例（揺れ） |
|----------|-------------|
| ニーズ (Need) | 課題、要望、ニード |
| ネタ (Idea) | アイデア、企画案（概要でのみ併記可） |
| サブネタ (SubIdea) | 小ネタ、サブアイデア |
| ソリューション (Solution) | 解決策 |
| 事例 (SolutionCase) | ケース、提案事例（概要でのみ併記可） |
| テナント (Tenant) | 組織、ワークスペース |
| お気に入り (NeedFavorite) | ブックマーク、スター |
| 議事録 (Source) | ソース、ミーティングノート |
| Server Action | サーバーアクション、SA |
| Route Handler | ルートハンドラー、API |

| スコア | 判定基準 |
|--------|----------|
| 0 | 同じ概念に異なる名前が使われている |
| 1 | 概ね統一だが、3箇所以上の揺れがある |
| 2 | 統一されている。軽微な揺れ1-2箇所 |
| 3 | 完全に統一。用語対応表がある |

### C6. アクセシビリティ

> 根拠: ISO/IEC/IEEE 26514:2022、Google Developer Documentation Style Guide -- ドキュメントは多様な読者・環境で利用可能であるべき

**チェック方法**:
1. Mermaid 図やテーブルに、色だけに依存した情報伝達がないか（色覚多様性への配慮）
2. 画像を使用している場合、代替テキスト（alt text）が付与されているか
3. テーブルが意味的に構造化されているか（ヘッダー行が明確、セル結合を避ける）
4. 長いコードブロックに目的や概要の説明文が付いているか

| スコア | 判定基準 |
|--------|----------|
| 0 | アクセシビリティへの配慮なし。色のみで区別する図がある |
| 1 | 基本的な構造化はされているが、alt text 欠落や色依存の箇所がある |
| 2 | テーブル・図とも構造的。色依存なし。一部 alt text 欠落 |
| 3 | 全画像に alt text。色以外の手段（ラベル、パターン）で情報伝達。スクリーンリーダーでも理解可能な構造 |

## カテゴリ D: 対象読者への適切さ

> 根拠: 「新規参画者もベテランも理解できるドキュメント」がゴール

### D1. 新規参画者への配慮

**チェック方法**:
1. `index.md` を読んだだけで「このシステムは何をするのか」が分かるか
2. ビジネスコンテキスト（B2B営業支援、議事録からニーズ抽出等）が説明されているか
3. 技術スタックの選定理由（なぜ Next.js か、なぜ Prisma か）は不要だが、何を使っているかは明示されているか
4. 「次に何を読むべきか」の導線があるか

**新規参画者のペルソナ**:
- Next.js / React の基礎知識はある
- このプロジェクトのドメイン（B2B営業、ニーズ管理）は知らない
- FSD アーキテクチャは初見かもしれない

| スコア | 判定基準 |
|--------|----------|
| 0 | ドメイン知識がないと理解不能 |
| 1 | 技術的には理解できるが、ビジネス目的が不明 |
| 2 | ビジネスコンテキスト + 技術概要があり、全体像を把握可能 |
| 3 | 上記 + 「次に何を読むか」の導線 + FSD の簡潔な説明 |

### D2. ユースケースの明示

**チェック方法**:
1. 各 feature doc に「この機能でできること」リストがあるか
2. 「ユースケース」セクションでビジネス上の利用場面が書かれているか
3. 技術的な「何ができるか」だけでなく「なぜ必要か」が伝わるか

| スコア | 判定基準 |
|--------|----------|
| 0 | ユースケース記述なし |
| 1 | 機能リストのみ。「なぜ」がない |
| 2 | 主要ユースケースが具体例付きで記述 |
| 3 | ビジネスユースケース + 技術的ユースケースの両方 |

### D3. オンボーディング効果

> 根拠: DORA 研究 -- ドキュメントの「Time-to-First-Success」がチームパフォーマンスに直結する

**チェック方法**:
1. `index.md` → 最初の feature doc → Server Action の呼び出し例まで、読者が迷わず到達できるか
2. 「開発環境のセットアップ → 最初のコード変更 → 動作確認」の一連の流れが文書化されているか
3. 前提知識の明示: 読者が何を知っている必要があるか（Next.js, Prisma, FSD 等）が冒頭で述べられているか
4. 「よくある最初のタスク」（例: 新しいニーズの作成）のウォークスルーがあるか

| スコア | 判定基準 |
|--------|----------|
| 0 | 初回タスク完遂までの導線がない |
| 1 | 技術スタックの説明はあるが、最初のタスクへの導線が不明確 |
| 2 | セットアップ手順 + 主要機能の概要があり、最初のタスクに着手可能 |
| 3 | セットアップ → 最初のタスク完遂までのウォークスルー。前提知識が明示。「次に何をすべきか」の導線あり |

### D4. 操作フローの可視化

**チェック方法**:
1. 各 feature doc に画面遷移図（Mermaid flowchart）があるか
2. データフロー（シーケンス図）が複雑な機能にあるか
3. ER 図がデータモデルセクションにあるか
4. Mermaid コードが正しい構文か（レンダリング可能か）

**Mermaid が必要な箇所**:
- `index.md`: 全体データフロー、ドメインモデル ER 図
- `needs.md`: 画面遷移、重複検出ロジック
- `ideas.md`: 画面遷移、Dify 連携フロー
- `auth.md`: 認証フロー、マルチテナントモデル
- `chat-consultation.md`: 通常モード/アポ設計モードのパイプライン
- `sources.md`: 議事録 → ニーズ抽出フロー
- `solutions.md`: 画面遷移、データモデル
- `analytics.md`: PV 記録フロー

| スコア | 判定基準 |
|--------|----------|
| 0 | 図なし |
| 1 | テキストでの説明のみ、または ASCII art のみ |
| 2 | 主要箇所に Mermaid 図がある |
| 3 | 全ての複雑なフローに Mermaid 図 + ロジック説明文。ER図でデータモデルを可視化 |

## カテゴリ E: リファレンス品質

> 根拠: Diataxis Reference -- リファレンスは網羅的・中立的・構造統一であるべき

### E1. API サーフェスの網羅性

**チェック方法**:
1. `app/api/` 配下の全 `route.ts` ファイルを列挙
2. 各 API が `routes.md` または対応する feature doc に記載されているか
3. Server Actions が入力/出力型付きで文書化されているか
4. HTTP メソッド（GET/POST/PATCH/DELETE）が正確か

**確認コマンド**: `find app/api -name "route.ts"` の結果と `routes.md` を照合

| スコア | 判定基準 |
|--------|----------|
| 0 | API の大半が未文書化 |
| 1 | コア API は文書化されているが、3つ以上の欠落がある |
| 2 | 全 API が文書化。入力型が記載 |
| 3 | 全 API が入力/出力型 + エラーケース付きで文書化 |

### E2. データモデルの文書化

**チェック方法**:
1. `prisma/schema.prisma` の model 一覧を取得
2. 主要モデル（Need, Idea, Solution, SolutionCase, SolutionIssue, ChatSession, Source, User, Tenant, TenantUser）がドキュメントに記載されているか
3. リレーション（1:1, 1:n, n:m）が図で表現されているか
4. 重要なインデックスやユニーク制約が記載されているか

| スコア | 判定基準 |
|--------|----------|
| 0 | データモデルの記述なし |
| 1 | テーブル名と主要カラムのみ |
| 2 | テーブル定義 + Mermaid ER 図 |
| 3 | テーブル定義 + ER 図 + ビジネスルール（論理削除、テナント境界等） |

### E3. 型定義のリファレンス

**チェック方法**:
1. 各 entity doc に主要な型定義（NeedForTable, IdeaWithRelations 等）が記載されているか
2. Zod スキーマ名が列挙されているか
3. 型のフィールドと説明があるか
4. 使用例（import 文 + 利用パターン）があるか

| スコア | 判定基準 |
|--------|----------|
| 0 | 型定義の記述なし |
| 1 | 型名の列挙のみ。フィールド不明 |
| 2 | 型定義のフィールドと説明がある |
| 3 | 型定義 + 使用例 + import パス + 関連 Zod スキーマ |

### E4. 構造の一貫性

**チェック方法**:
1. `_template.md` のセクション構成を確認
2. 各 feature doc がテンプレートに沿っているか比較
3. セクションの順序が統一されているか（概要 → ルーティング → データモデル → Server Actions → UI → 状態管理 → 認証 → 実装場所 → 操作フロー → 関連機能）

| スコア | 判定基準 |
|--------|----------|
| 0 | セクション構成がバラバラ |
| 1 | 一部がテンプレートに従うが、3つ以上のドキュメントで構成が異なる |
| 2 | 概ね統一。軽微な順序の違いのみ |
| 3 | 全 feature doc がテンプレートに準拠。セクション順序が統一 |

## カテゴリ F: 保守性

> 根拠: Write the Docs Docs-as-Code -- ドキュメントもコードと同じワークフローで管理

### F1. コードとの近接性

**チェック方法**:
1. ドキュメントが同一リポジトリの `docs/next/` に配置されているか
2. `CLAUDE.md` の「ドキュメント更新ルール」が参照可能か
3. `docs-auditor` エージェントが設定されているか

| スコア | 判定基準 |
|--------|----------|
| 0 | 別システムで管理 |
| 1 | 同リポジトリだが更新プロセスがない |
| 2 | 同リポジトリ + `CLAUDE.md` に更新ルール記載 |
| 3 | 同リポジトリ + 更新ルール + `docs-auditor` による自動検出 |

### F2. 単一情報源（Single Source of Truth）

**チェック方法**:
1. 同じ情報（例: safe-action ミドルウェアチェーン）が複数ファイルに重複していないか
2. 重複がある場合、一方がもう一方を「詳細: [Lib](../lib/index.md)」のように参照しているか
3. `index.md` と各 feature doc で情報が矛盾していないか

**重複しやすい情報**:
- safe-action ミドルウェアの説明（lib/index.md, features/index.md, 各 feature doc）
- 認証パターン（auth.md, 各 feature doc の認証セクション）
- テナント境界チェック（auth.md, needs.md, ideas.md 等）

| スコア | 判定基準 |
|--------|----------|
| 0 | 同一情報が多数箇所に重複。矛盾あり |
| 1 | 重複はあるが矛盾はない |
| 2 | 重複最小限。重複箇所は相互参照で繋がっている |
| 3 | 各事実は1箇所で定義。他は全てリンク参照 |

### F3. テンプレートと規約

**チェック方法**:
1. `_template.md` が存在するか
2. テンプレートのセクション構成が実際のドキュメントと一致するか
3. 新規 feature 追加時にテンプレートから始められるか

| スコア | 判定基準 |
|--------|----------|
| 0 | テンプレートなし |
| 1 | テンプレートはあるが実態と乖離 |
| 2 | テンプレートが存在し、大半のドキュメントが準拠 |
| 3 | テンプレート + 「新規 feature のドキュメント作成手順」が明確 |

### F4. 陳腐化の検出

**チェック方法**:
1. `docs-auditor` / `docs-freshness-auditor` エージェントが設定されているか
2. `CLAUDE.md` のドキュメント更新ルールが機能しているか
3. 自動検出の仕組みが存在するか

| スコア | 判定基準 |
|--------|----------|
| 0 | 検出の仕組みなし |
| 1 | 手動レビューのみ |
| 2 | `docs-auditor` が設定されている |
| 3 | `docs-auditor` + `docs-freshness-auditor` + `CLAUDE.md` の自動更新ルール |

### F5. フィードバック機構

> 根拠: Write the Docs コミュニティ推奨 -- ドキュメントの品質改善ループを閉じるための手段

**チェック方法**:
1. ドキュメントの誤りや改善要望を報告する手段があるか（Issue テンプレート、連絡先等）
2. 「このページは役に立ちましたか？」のような定性フィードバック手段があるか
3. フィードバックが実際にドキュメント改善に反映されるプロセスがあるか

| スコア | 判定基準 |
|--------|----------|
| 0 | フィードバック手段なし |
| 1 | リポジトリの Issue で報告可能だが、ドキュメント内に案内がない |
| 2 | ドキュメント内にフィードバック先（Issue リンク等）が明示されている |
| 3 | フィードバック先の明示 + フィードバックを反映するプロセス（トリアージルール等）が定義されている |

## 集計方法

### カテゴリ別スコア

| カテゴリ | 基準数 | 満点 | 重み | 意味 |
|----------|--------|------|------|------|
| A. 構造と構成 | 5 | 15 | 必須 | スコア0の基準があれば総合評価を制限 |
| B. コンテンツ品質 | 5 | 15 | 必須 | スコア0の基準があれば総合評価を制限 |
| C. 文章品質 | 6 | 18 | 重要 | 優先的に改善すべき領域 |
| D. 対象読者への適切さ | 4 | 12 | 重要 | 優先的に改善すべき領域 |
| E. リファレンス品質 | 4 | 12 | 必須 | スコア0の基準があれば総合評価を制限 |
| F. 保守性 | 5 | 15 | 推奨 | 長期的に改善していく領域 |
| **合計** | **29** | **87** | | |

**重みの運用**: 重みは加重計算には使用しない。改善の優先順位付けに使う。「必須」カテゴリの低スコア基準は「推奨」カテゴリよりも先に対処する。

### 総合評価

| スコア範囲 | 評価 | 意味 |
|------------|------|------|
| 0-21 (0-24%) | **不十分** | 基本的な文書化が不足。即時対応が必要 |
| 22-43 (25-49%) | **要改善** | 基盤はあるが多くの改善が必要 |
| 44-65 (50-74%) | **合格** | 実用上の問題はない。段階的に改善 |
| 66-78 (75-89%) | **良好** | ベストプラクティスに概ね準拠 |
| 79-87 (90-100%) | **優秀** | 業界標準の高品質ドキュメント |

### ブロッカー条件

以下のいずれかに該当する場合、総合スコアに関わらず「要改善」とする:
- B1 (正確性) が 0 -- 事実誤認のあるドキュメントは害悪
- A3 (情報アーキテクチャ) が 0 -- 構造がないと全体が成立しない
- D1 (新規参画者) が 0 -- 誰も読み始められない

## レビュー出力テンプレート

レビュアー（人間/AI）は以下のフォーマットで結果を出力する:

```markdown
## レビュー結果: [レビュー観点名]

### スコアサマリー

| 基準 | スコア | 根拠（1文） |
|------|--------|------------|
| A1 | 2 | リファレンスとハウツーは分離されているが、説明セクション内にリファレンスが混入 |
| A2 | 3 | 全ドキュメントに相互リンクあり。孤立ページなし |
| ... | ... | ... |

### カテゴリ別スコア

| カテゴリ | スコア | 満点 |
|----------|--------|------|
| A | 12 | 15 |
| ... | ... | ... |
| **合計** | **XX** | **87** |

### 検出された問題（重要度順）

重要度の判定基準:
- **Critical**: 「必須」カテゴリでスコア0、またはブロッカー条件に該当
- **High**: 「必須」カテゴリでスコア1、または事実誤認・デッドリンク
- **Medium**: 「重要」カテゴリでスコア1以下、または一貫性の欠如
- **Low**: 「推奨」カテゴリの改善点、または軽微なフォーマット問題

1. **[Critical]** 問題の説明 -- 該当ファイル:行
2. **[High]** ...
3. **[Medium]** ...
4. **[Low]** ...

### 改善提案（優先度順）

1. 提案内容 -- 期待される効果
2. ...
```

## レビュー実行ガイド

### マルチエージェントレビューの推奨構成

| エージェント名 | 担当カテゴリ | 指示のポイント |
|---------------|-------------|---------------|
| structure-reviewer | A (構造) | ドキュメント間のリンクと階層構造を検証 |
| accuracy-reviewer | B1, E1-E3 (正確性・リファレンス) | コードベースと照合。型/パス/バージョン |
| writing-reviewer | C (文章品質) | 用語統一、文体統一、フォーマット統一 |
| audience-reviewer | D (読者適切性) | 新規参画者の視点でindex.mdから読み始める |
| freshness-reviewer | B3, F (鮮度・保守性) | git log との照合、重複検出、仕組みの確認 |
| visualization-reviewer | D4 (可視化) | Mermaid 構文の正当性、図の網羅性 |
| accessibility-reviewer | C6 (アクセシビリティ) | alt text、色依存、構造的マークアップの確認 |

各エージェントには:
1. この評価基準ファイルを読ませる
2. 担当カテゴリの「チェック方法」に従わせる
3. 「レビュー出力テンプレート」で結果を返させる
