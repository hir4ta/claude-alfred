# JARVIS Score: claude-buddy 採点ルーブリック

> Claude Code (Tony) の相棒 claude-buddy (JARVIS) の性能を客観的に評価する。
> 56 基準 x 7 カテゴリ = 100 点満点。

## 採点方法

各基準を 0-5 で採点。カテゴリスコア = (基準スコアの合計 / カテゴリ内の満点) x カテゴリ配点。

| カテゴリ | 配点 | 基準数 | 満点(raw) |
|---------|------|--------|-----------|
| A. 予見力 (Anticipation) | 20 | 10 | 50 |
| B. シグナル品質 (Signal Quality) | 20 | 9 | 45 |
| C. 学習能力 (Learning) | 15 | 8 | 40 |
| D. インターフェース (Interface) | 15 | 8 | 40 |
| E. 安全防護 (Safety) | 10 | 7 | 35 |
| F. 初回体験 (Cold Start) | 10 | 7 | 35 |
| G. セッション横断 (Continuity) | 10 | 7 | 35 |

---

## A. 予見力 (20 点)

*JARVIS の核心: Tony が行動する「前」に選択肢を提示し、最適解へ導く。*

---

### A1. 行動前の選択肢提示

**ツール実行前に、根拠付きの代替案を番号付きで提示しているか？**

| Score | 定義 |
|-------|------|
| 0 | 代替案を一切提示しない |
| 1 | 1-2 のハードコードシナリオでのみ警告（force-push のみ等） |
| 2 | Edit/Write の失敗履歴に基づいて代替案を提示（stale read、高失敗率） |
| 3 | 複数ツール種別で代替案を提示（Edit/Bash/Read/Grep/Task）。ツール固有の根拠付き。最大3案、優先度ソート済み |
| 4 | 過去セッションのエビデンス（resolution diff、co-change、アーキテクチャパターン、ツールシーケンス予測）を含む。各案に "Why:" の深い理由付き |
| 5 | セッションコンテキスト（task type, phase, domain, user profile）で動的ランク。クロスプロジェクトエビデンス付き。ユーザーの選択を50%以上の精度で予測 |

---

### A2. エピソード早期警告

**アンチパターンが完全に形成される「前」に検出しているか？**

| Score | 定義 |
|-------|------|
| 0 | エピソード検出なし |
| 1 | 完全形成後に1パターン検出（3回以上のリトライ等） |
| 2 | 2-3パターンを早期段階で検出（60-70%マッチ閾値） |
| 3 | 5つのハードコードテンプレート + 部分一致。ドメイン別アドバイス。クールダウンでアラート疲労防止 |
| 4 | ハードコード + 過去セッションから動的学習したエピソード。軌道マッチング（Jaccard類似度） |
| 5 | シーケンスモデルによるリアルタイム予測。40%一致で信頼度付き発火。新エピソード形状を毎セッション自動学習。偽陽性率10%未満 |

---

### A3. プロアクティブな解決策提示

**ユーザーが同じエラーに遭遇する「前」に、過去の解決策を浮上させているか？**

| Score | 定義 |
|-------|------|
| 0 | プロアクティブな解決策提示なし |
| 1 | エラー発生後にのみ解決策を提示 |
| 2 | UserPromptSubmit で直近の失敗を解決策DBと照合 |
| 3 | PreToolUse で未解決失敗のファイルに対し resolution diff + solution chain を検索。`old -> new` 形式 + ステップ数で表示 |
| 4 | resolution diff + solution chain + マルチステップ playbook を統合。failure type / error signature / file path の3軸検索。提示した solution ID を追跡し効果測定 |
| 5 | embedding による意味的解決策マッチング。ファイル + アクションの組み合わせから「どのエラーが起きるか」を予測し、先回りで解決策を提示。再発防止効果を計測 |

---

### A4. フェーズ遷移ガイダンス

**ワークフローフェーズの遷移を検出し、プロアクティブなコーチングを提供しているか？**

| Score | 定義 |
|-------|------|
| 0 | フェーズ認識なし |
| 1 | read/write/test/compile フェーズを追跡するが助言なし |
| 2 | フェーズ遷移を検出し汎用テンプレートで助言 |
| 3 | 8タスクタイプ x 2-5フェーズの具体的コーチング。ドメインオーバーライド（auth/database/infra/api/ui）。リスクシナリオでの事前コーチング |
| 4 | クロスセッションデータでパーソナライズ（成功/失敗セッションのツール数比較、テスト頻度）。ユーザークラスタ別トーン調整。カバレッジマップからテストコマンド注入 |
| 5 | AI生成コーチングが現セッションの具体的ファイルと決定を参照。コーチング効果を追跡しテンプレートを進化。クラスタではなく個人の弱点に適応 |

---

### A5. 速度壁予測

**生産性低下を発生「前」に予測しているか？**

| Score | 定義 |
|-------|------|
| 0 | 速度追跡なし |
| 1 | EWMA velocity を追跡するが現在値の報告のみ |
| 2 | 速度壁を発生「後」に検出（50%以上の低下） |
| 3 | EWMV ベースの適応的制御限界。mean +/- k*sigma で壁検出 |
| 4 | OLS 回帰によるヘルストレンド予測 + 勾配外挿。閾値到達 N ツール前に警告。分散閾値（ewmv > 4.0）でゲート |
| 5 | 速度 + エラー率 + フェーズ分布 + 過去パターンの多要因予測。過去の壁エピソードに基づく回復時間推定。具体的回復 playbook を提供 |

---

### A6. 共変更予測

**一緒に変更すべきファイルを予測しているか？**

| Score | 定義 |
|-------|------|
| 0 | ファイル関係の認識なし |
| 1 | 同一セッションで変更されたファイルを追跡 |
| 2 | セッション横断で共変更カウントを保存。Y 編集時に "also check X" を表示 |
| 3 | PreToolUse と JARVIS briefing (P3) の両方で共変更ヒント。working set 比較で欠落パートナーを特定。クールダウン制御（10-15分） |
| 4 | セッション数による重み付き共変更ペア（最低閾値3セッション）。代替案に "Co-changed file" + Why 説明。グローバルストアで永続化 |
| 5 | 依存グラフベースの共変更予測（歴史的相関だけでなく構造的結合）。偶発的共変更と構造的共変更を区別。共変更ファイルに「何を変更すべきか」まで予測 |

---

### A7. 意図認識ガイダンス

**ユーザーが「何を達成しようとしているか」を理解し、それに沿ったガイドをしているか？**

| Score | 定義 |
|-------|------|
| 0 | 意図認識なし |
| 1 | 単一のタスクタイプラベルに分類 |
| 2 | 8タスクタイプ（bugfix/feature/refactor/test/explore/debug/review/docs）をプロンプトキーワードで分類。working set に保存 |
| 3 | 意図遷移の検出。ドメイン分類（auth/database/ui/api/config/infra/test）。タスクタイプで知識タイプの優先度・コーチング・検出閾値を調整 |
| 4 | 意図が代替案（bugfix without tests → "Run tests first"）、コーチング（phase x task type マトリクス）、解決策検索の優先度を駆動。ドメイン別事前コーチング。短いプロンプト時のファイルパス推論フォールバック |
| 5 | 深い意図理解: マルチステップタスクをサブゴールに分解、各サブゴールの進捗を追跡、完了に応じてガイダンスを調整。コード変更からの意図推論（プロンプトテキストだけでなく） |

---

### A8. 変更前の影響分析

**編集前にブラストラディウスと下流への影響を分析しているか？**

| Score | 定義 |
|-------|------|
| 0 | 影響分析なし |
| 1 | 編集中のファイル名を表示 |
| 2 | 編集中ファイルの importer を一覧表示 |
| 3 | ブラストスコア (0-100) + リスクレベル + importer 数 + テストファイル発見。ファイル毎クールダウン。PreToolUse とスタートアップで表示 |
| 4 | 依存グラフ走査 + カバレッジマップによる標的テスト提案 + ドメインリスク警告（auth, database, infra, api） |
| 5 | 推移的影響分析: 完全な依存グラフを辿って下流消費者を追跡。各影響ファイルの過去の失敗率に基づくリグレッション確率推定。影響ゾーン毎の具体的テストコマンド |

---

### A9. タスク工数見積もり

**過去履歴に基づいてタスクの規模/複雑さを予測しているか？**

| Score | 定義 |
|-------|------|
| 0 | 見積もり機能なし |
| 1 | 過去セッションのツール数のみ |
| 2 | 成功ワークフローから中央値ツール数と成功率を算出（タスクタイプでフィルタ） |
| 3 | MCP ツール（buddy_estimate）が中央値ツール数・成功率・一般的ワークフローパターンを返却。セッションペース洞察が現在進捗を過去中央値と比較（1.5x閾値） |
| 4 | 戦略プラン（buddy_strategic_plan）がフェーズ毎工数見積もり付きの最適フェーズシーケンスを生成（ユーザースタイル適応）。タスク進捗（buddy_task_progress）がマルチセッションチェーンとワークフローパターンを表示 |
| 5 | 信頼区間付き見積もり。ファイル複雑度（LOC, cyclomatic complexity）を考慮。セッション進行に伴いリアルタイム更新。見積もり精度を追跡・表示 |

---

### A10. ワークフロー逸脱検出

**現在のワークフローが過去の成功パターンから逸脱した時に検出しているか？**

| Score | 定義 |
|-------|------|
| 0 | ワークフローパターン追跡なし |
| 1 | ツールシーケンスを記録するが比較なし |
| 2 | 成功ワークフローシーケンスを保存。基本的な bigram ツール予測 |
| 3 | trigram + bigram の失敗予測（セッション + グローバルデータ統合）。現在のフェーズシーケンスを最も一般的な成功ワークフローと比較して逸脱警告 |
| 4 | フェーズ bigram の Jaccard 類似度で過去の失敗セッションと軌道マッチング。working set にワークフローフェーズと次の想定フェーズを付加 |
| 5 | ツール呼び出し毎に更新されるオンラインワークフロー整合スコア。成功軌道と失敗軌道の「同時」検出。逸脱検出時に正確な分岐点を参照した具体的修正アクションを提案 |

## B. シグナル品質 (20 点)

*助言は具体的か？実行可能か？文脈に合っているか？汎用テンプレートではないか？*

---

### B1. 提案の具体性

**提案は追加調査なしでそのまま実行できるほど具体的か？**

| Score | 定義 |
|-------|------|
| 0 | 提案なし |
| 1 | 汎用ヒント: "テストの実行を検討してください" |
| 2 | ファイルまたはツール名を含む: "handler.go のテストを実行" |
| 3 | コピペ可能なコマンド: "Run: go test -run TestHandler ./internal/handler/"。resolution diff で `old -> new` の具体コード |
| 4 | カバレッジマップから正確なテスト関数マッピングによるコマンド導出。ステップ数付きファイル固有の解決チェーン。ファイル毎の未テスト関数リスト |
| 5 | 完全な playbook: 各ステップの期待出力付き順序化コマンドシーケンス。具体的エラーメッセージとファイル状態に基づくコマンド適応。ロールバック手順付き |

---

### B2. 文脈の適切性

**シグナルは汎用ルールではなく、現在のセッション文脈を使用しているか？**

| Score | 定義 |
|-------|------|
| 0 | 全助言が静的/汎用 |
| 1 | ツール名でアドバイスカテゴリを選択 |
| 2 | task type + phase でコーチングテンプレートを選択 |
| 3 | working set（ファイル、intent、domain、git branch）、失敗履歴、burst state（ツール数、read/write状態）、EWMAメトリクスを使用 |
| 4 | クロスセッションデータ: user profile メトリクス（test_frequency, read_write_ratio）、成功ワークフロー比較、recurring struggle パターン、Thompson Sampling でこのユーザーの応答パターンに基づく優先度調整 |
| 5 | 全シグナルがライブセッション状態 + クロスセッション学習 + クロスプロジェクトパターンの統合。テンプレートからは生成不可能な助言。各シグナルがユーザーの実際の履歴から具体的データを参照 |

---

### B3. シグナル対ノイズ比

**配信されるシグナルのうち、どの程度が実際にアクションにつながるか？**

| Score | 定義 |
|-------|------|
| 0 | 大半のシグナルが無視/無関係 |
| 1 | クールダウンで完全重複を防止するが、大半は無視される |
| 2 | Thompson Sampling が非効果的パターンを降格。過剰配信・低解決パターンの基本的抑制 |
| 3 | フローステート分類（Normal/Productive/Thrashing/Stalled/Fatigued）で段階的抑制。ユーザークラスタ別バーストキャップ（conservative:5, balanced:3, aggressive:1）。文字数バジェット（2000）で優先度ソートしたシグナルフィッティング |
| 4 | パターン固有の減衰半減期（tactical 14d, strategic 60d）。KL正則化で過学習防止。explicit feedback が auto-feedback をオーバーライド。フェーズゲーティング。ワークフロー境界ブースト。繰り返しパターンのエンリッチメント |
| 5 | 測定されたSNRが80%超（配信シグナルの80%超がアクションにつながる）。3セッション以内にノイズソースを自動特定・除去。ユーザーが抑制を設定する必要なし |

---

### B4. タイミングの精度

**シグナルは正しいタイミングで届くか — 早すぎず、遅すぎず？**

| Score | 定義 |
|-------|------|
| 0 | ランダムなタイミング |
| 1 | シグナルは届くが、多くは手遅れ（エラー発生後） |
| 2 | 全 UserPromptSubmit（ターン開始時）+ 一部 PreToolUse |
| 3 | PreToolUse でリスキーなアクション直前に発火。エピソード早期警告が60-67%パターン完成時に発火。JARVIS briefing がユーザーターン毎に最重要シグナル1つ |
| 4 | ワークフロー境界ブーストがフェーズ遷移・git commit・タスク切替時に発火。事前コーチングが初回書き込み前・壊れたビルドでのテスト前・ドメイン敏感操作前に発火。速度壁警告が閾値前に発火 |
| 5 | 知覚最適化タイミング: 最大受容性の瞬間（自然な一時停止ポイント）にシグナル。非クリティカル情報でフローを中断しない。タイミング自体がユーザークラスタに基づき適応 |

---

### B5. 説明の深さ (Why)

**シグナルは「何をすべきか」だけでなく「なぜ重要か」を説明しているか？**

| Score | 定義 |
|-------|------|
| 0 | 説明なし |
| 1 | 理由なしの一行ヒント |
| 2 | コーチングに "WHY:" 行で一般的理由 |
| 3 | 3部構造: situation + reasoning + suggestion。代替案に "Why:" 行でツール固有の説明 |
| 4 | 推論にパーソナルデータ: "あなたの成功 bugfix セッションは平均22ツール"。ドメイン固有推論。エビデンスベースの主張（"解決時間60%長期化"） |
| 5 | 因果チェーン推論: "X が起きた → Y が起きやすい、なぜなら Z → したがって W をすべき"。このユーザーの過去の具体的インシデントを参照。推論の信頼度を定量化 |

---

### B6. 優先度判定の正確性

**「今、最も重要なシグナル」を正しくランク付けしているか？**

| Score | 定義 |
|-------|------|
| 0 | 優先度なし — 全て同等 |
| 1 | Critical/non-critical の二値分割 |
| 2 | 4段階優先度（Critical/High/Medium/Low）静的割り当て |
| 3 | Thompson Sampling で delivery/resolution データに基づく優先度調整。JARVIS briefing で7段階優先度ラダー（P0-P6）。シグナルバジェットソートで最高優先度が文字数制限内に収まる |
| 4 | コンテキスト付き Thompson Sampling (pattern:taskType:cluster) + KL正則化。explicit feedback が自動推論をオーバーライド。ドメインとフェーズで不適切シグナルをゲート。ワークフロー境界ブースト |
| 5 | 優先度の正確性を測定し85%超（配信された最高優先度シグナルがユーザーが行動するもの）。要求時に優先度判定の理由を説明。セッション内でユーザーアクションに基づき再キャリブレーション |

---

### B7. ドメイン感度

**問題領域（auth, database, API等）に基づいてアドバイスが適応しているか？**

| Score | 定義 |
|-------|------|
| 0 | ドメイン認識なし |
| 1 | ドメインは検出されるがアドバイスに使用されない |
| 2 | ドメインでコーチングテンプレートを選択（1-2ドメイン） |
| 3 | 6ドメイン（auth/database/ui/api/config/infra）でドメイン固有コーチングオーバーライド、エピソードアドバイス、事前コーチングトリガー、リスク警告 |
| 4 | プロンプトキーワード + ファイルパス推論フォールバックでドメイン分類。ツールタイプ毎のドメインリスクチェック。複数 task type x phase で ドメインコーチング |
| 5 | フレームワーク固有ベストプラクティス（Django ORM パターン、React hook ルール等）。ASTベースのドメイン検出（キーワードだけでなく）。ドメイン固有テスト戦略とデプロイチェックリスト |

---

### B8. ネガティブアドバイス（やめるべき時）

**Claude Code に「止まれ」「それをするな」と伝えているか？**

| Score | 定義 |
|-------|------|
| 0 | "やめろ" とは言わない |
| 1 | 破壊的コマンドをブロック（rm -rf, git push --force） |
| 2 | ブロック + リスキー操作の警告。retry_cascade と edit_fail_spiral でのエピソード deny |
| 3 | 高失敗率ゲート: 特定ファイルで80%以上の失敗率（5回以上）で確認要求。デバッグスパイラルでの "一歩引いて"。trigram 予測失敗での "別アプローチを検討" |
| 4 | フェーズ適切なネガティブアドバイス: "壊れたビルドでテストを実行するな"、"テストファイルを読まずに編集するな"、"再現なしでバグ修正を委任するな"。ドメイン固有: "環境チェックなしでデプロイするな" |
| 5 | 定量化されたネガティブアドバイス: "ここで止まったことで過去3セッションで15ツール節約"。ネガティブアドバイスを無視した場合の実際のコストを追跡。"X をするな" だけでなく代替パスを提供 |

---

### B9. マルチシグナル統合

**複数のシグナルを一貫したナラティブに統合できるか？**

| Score | 定義 |
|-------|------|
| 0 | 各シグナルが独立・孤立 |
| 1 | シグナルを順番に連結 |

## 付録: Claude Code 公式ベストプラクティス準拠チェック

*claude-buddy が Claude Code のエクステンションポイント（MCP / Hooks / Skills）の公式ベストプラクティスにどの程度準拠しているかを評価するチェックリスト。スコアには含めないが、改善時の設計指針として参照。*

---

### BP1. MCP ツール数・出力制限・Tool Search 対応

| 項目 | ベストプラクティス |
|------|------------------|
| ツール定義のコンテキスト消費 | 全ツール定義がコンテキストの10%超 → Tool Search 自動有効化。Haiku 非対応に注意 |
| server_instructions | Tool Search 有効時に "このサーバーの役割" を説明する instructions が必須 |
| ツール数の方針 | 目的が明確に異なるツールのみ。機能重複は統合（"consolidate"）。公式の具体的上限数はないが、少ないほどモデルの選択精度が向上 |
| ツール名の制約 | 最大64文字。正規表現 `^[a-zA-Z0-9_-]{1,64}$` に準拠 |
| MCP 出力サイズ | 10,000トークン超で Claude Code が警告表示。25,000トークンがハードリミット（超過分は切り捨て） |

---

### BP2. MCP ツール説明の品質

| 項目 | ベストプラクティス |
|------|------------------|
| 必須要素（3-4文） | (1) 何をするか (2) いつ使うか (3) 主要パラメータの効果 (4) 重要な注意点や制約 |
| 書式 | 3人称記述（"Searches..." "Returns..."）。"I can..." は不可 |
| トリガー条件 | "Use when..." / "Call when..." で使用タイミングを明示 |
| パラメータ説明 | 各パラメータに description + デフォルト値を記載 |
| ツール名の制約 | 最大64文字。正規表現 `^[a-zA-Z0-9_-]{1,64}$`。ハイフンまたはアンダースコアで単語区切り |

---

### BP3. MCP パラメータスキーマ

| 項目 | ベストプラクティス |
|------|------------------|
| description 付き | 全パラメータに description 必須 |
| required 明示 | 必須/任意を明確に |
| デフォルト値記載 | description にデフォルト値を記載 |
| outputSchema | 構造化レスポンスのスキーマ定義 |

---

### BP4. MCP ツール Annotations

| 項目 | ベストプラクティス |
|------|------------------|
| readOnlyHint | 読み取り専用ツールに `true` |
| destructiveHint | 書き込みツールに適切に設定 |
| idempotentHint | 冪等ツールに `true` |
| openWorldHint | 外部API呼び出しツールに設定 |
| title | 人間可読な表示名 |

---

### BP5. Hook レスポンスフィールドと終了コード

| 項目 | ベストプラクティス |
|------|------------------|
| 終了コード | `0` = 成功（JSON 出力を処理）。`2` = ブロック（操作中断を指示）。その他 = 非ブロッキングエラー（stderr ログ、処理継続） |
| ユニバーサルフィールド | `continue`（会話継続制御）、`stopReason`（停止理由）、`suppressOutput`（出力抑制）、`systemMessage`（システムメッセージ注入） |
| PreToolUse | hookSpecificOutput 内に permissionDecision（allow/deny/ask）+ permissionDecisionReason。updatedInput でサイレント修正 |
| PostToolUse | 非同期フック → トップレベル additionalContext |
| UserPromptSubmit | 同期フック → hookSpecificOutput 内 additionalContext |
| Stop | トップレベル Decision（"block"）+ Reason。ブロック時は exit code 2 も使用可 |
| 空レスポンス処理 | 出力なし時は nil 返却（stdout に何も書かない） |

---

### BP6. Hook サイズ制約・タイムアウト・フックタイプ

| 項目 | ベストプラクティス |
|------|------------------|
| フックタイプ | `command`: シェルコマンド実行。`prompt`: Claude にプロンプト注入。`agent`: サブエージェントスポーン |
| 公式デフォルトタイムアウト | command: 600秒 / prompt: 30秒 / agent: 60秒。各フックに `timeout` フィールドで秒数指定可 |
| PreToolUse タイムアウト | ツール実行をブロックするため高速必須。claude-buddy は 2秒に設定（独自判断） |
| PostToolUse | `async: true` で非同期配信。結果は次のターンで Claude に到着 |
| SessionStart | バックグラウンド処理を活用し体感遅延を最小化 |
| additionalContext サイズ | 簡潔推奨。コンテキストウィンドウを消費するため必要最小限に |

### BP7. Skills 設計

| 項目 | ベストプラクティス |
|------|------------------|
| SKILL.md 行数 | 500行以下推奨（公式上限） |
| SKILL.md 語数 | 5,000語以下推奨 |

### BP8. Memory 機能との連携

| 項目 | 活用方法 |
|------|---------|
| auto memory ディレクトリ | `~/.claude/projects/<project>/memory/` にプロジェクト固有の学習結果を書き込み |
| MEMORY.md | 200行以内で常にコンテキストにロード。セッション横断の重要知識をここに |
| トピック別ファイル | `memory/patterns.md`, `memory/decisions.md` 等でリンク |
| Hook 連携 | SessionEnd で学習結果を memory に書き出し。SessionStart で既存 memory を読み込み |
| MCP ツール削減効果 | 常にコンテキストにある情報は MCP ツール不要 → ツール数削減に直結 |

#### Memory 活用の具体的アイデア

```
~/.claude/projects/<project>/memory/
  MEMORY.md          ← 常にロード（200行以内）
    - プロジェクト概要（buddy が学習した構造）
    - 直近の重要決定 top 5
    - よくある失敗パターン top 3 + 解決策
    - ユーザーの作業スタイル要約
    - buddy 機能の使い方ガイド（MCP ツール数削減後の簡潔版）
  patterns.md        ← 詳細パターン集（MEMORY.md からリンク）
  recent-session.md  ← 前セッションの要約（resume の代替）
```

**最大の利点**: MCP ツールを呼ばなくても、毎セッション自動的にバディの知識がClaude Codeのコンテキストに入る。「聞かれる前に知っている」JARVIS の理想形。
